# v0.4 Work Plan — Project Structure

Prepared and first slice landed 2026-07-08. Scope from the roadmap:
multi-file programs with merge semantics, AZM module imports, and the
dependency listing.

## Landed in the first slice

1. **`part "file.glim"`** — the entry file declares the program,
   platform, and display, and names its parts; every part contributes
   declarations to one program and one namespace. Merge semantics, not
   inclusion: parts may not declare `program`, `platform`, `display`,
   or nested `part`s. Paths resolve relative to the entry file.
   Diagnostics are file-tagged (`blocks.glim:12: Duplicate name ...`).
   Demonstrated by `examples/trail.glim` + `trail-blocks.glim`.
2. **`import "module.asm"`** — brings a hand-written AZM module into the
   generated program. Emitted as `.import` in a dedicated section after
   the frame rollover, outside every execution path; `@` labels are the
   module's public API and its plain labels stay private to the unit.
3. **`glimmer --deps entry.glim`** — the dependency report: for every
   flag-carrying cell, who raises it (keys, timers, ramps, blocks) and
   which blocks it triggers, with phases. The program's dataflow without
   reading any Z80.

## Deferred within v0.4

- **Generated output as `.import` modules** (block privacy beyond the
  naming convention): needs a file-layout decision — one module per
  part? per card? — better taken with v0.5 structured data in view.
- **Per-block assemble/check**: an editor-era feature; revisit with the
  Debug80 integration work.
- **`.glim` libraries**: still deferred pending a namespace story.

## Notes

- `parseGlimmer(text)` (the single-file API) reports `part` declarations
  as an error pointing at the CLI/loader; `loadGlimmerProgram(entryPath)`
  is the multi-file entry point, exported from the package.
- Import bytes land at the directive: the dedicated section placement is
  load-bearing, not cosmetic.
